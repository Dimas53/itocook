Layer breakdown — what's inside each block
1 · INSTRUCTION
Rules of the Game
The foundation — everything the agent reads before doing anything
AGENTS.md
Main rules file — safety gates, workflow rules, what to never do
Trigger-based docs
Agent updates docs only on specific events — new composable, new Flow, bug fix
Session scope rules
One concern per session, verify in browser before commit, plan before build
2 · CONTEXT
What the Model Sees
Documents loaded into context window each session
progress.md
Living log of what's done, current bugs, session fixes
roadmap.md
Big picture phases — agent reads this to not dive into wrong phase
design.md
Single source of truth for all UI decisions — read before any component
3 · TOOLS
MCP Servers
What the agent can actually do — files, git, browser, APIs
filesystem MCP
Read and write any project file — components, configs, docs
git MCP
Commits, history, diffs — agent tracks its own changes
Directus MCP
Full admin access — creates collections, flows, sets permissions autonomously
Chrome DevTools
Reads browser console, network, errors — debugs without screenshots
Playwright MCP
E2E tests, form filling, navigation, screenshots — full browser test runner
4 · LOOP
Execution Cycle
Think → Do → Verify — the discipline that prevents cascading failures
Plan before building
Multi-file tasks get a plan first — approved before any code written
One change at a time
Learned the hard way — multiple simultaneous infra changes = full-day rollback
Verify in browser first
Agent checks result in DevTools before committing anything to git
5 · MEMORY
Notes Between Steps
Trigger-based — agent writes docs when specific events happen, not on schedule
progress.md
Updated on every bug fix, feature complete, or session end
architecture.md
Updated when new composable, route, or public interface is added
docs/specs/
Feature specifications — created for new features without existing specs
git log
Structured commit messages act as searchable history of decisions
6 · SUBAGENTS
Intelligence Tools
Specialized models and tools called for specific subtasks
context7
Pulls live docs for Nuxt, Directus, Vue — always gets current API, not training data
sequential-thinking
Forces step-by-step reasoning before jumping to implementation
7 · VERIFICATION
Quality Gates
Checks that run before any commit is made
TypeScript check
Zero TS errors rule — agent fixes all type errors before committing
Browser DevTools
Console errors, network 4xx/5xx, CORS — checked via Chrome DevTools MCP
Screenshot review
Visual confirmation — Dmitrii catches UI regressions the agent misses
Test suite (planned)
4 Vitest unit + 3 Vitest API + 3 Playwright E2E + CI/CD pipeline — 11 prompts in notes/tests-promt.md
8 · SANDBOX
Boundaries & Safety Gates
Hard rules that require human approval before proceeding
nginx changes require approval
Learned after a config bug caused full-day outage — now explicit gate
Service Worker changes require approval
SW strategy switches affect all cached users — can't be reversed without full cache clear
No production secrets in prompts
Real tokens, passwords, DB strings stay out of Claude/agent conversations
9 · SKILLS
Knowledge Loaded on Demand
Agent reads relevant skill files before starting any task in that domain
security/ — 5 files
Custom security baseline: auth, API, frontend, stack-specific, release checklist
session-start
Agent boot sequence — reads docs, confirms scope, checks for blockers before coding
code-reviewer
PR review rules — checks for common patterns, security issues, doc updates
codebase-health-check
Architecture audit skill — assesses codebase, finds refactoring opportunities, DRY violations
nuxt · directus · docker · tailwind
Public stack skills — best practices, common pitfalls, API patterns for each tool
make-interfaces-feel-better
UI polish skill — animations, micro-interactions, visual details for final phase
