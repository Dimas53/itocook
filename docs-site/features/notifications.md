# Notifications & Push

## What It Does

Manages the full notification lifecycle: in-app notifications stored in Directus, push notifications delivered via Service Worker, and schedule-based reminders. Users receive alerts for cooking assignments, lunch readiness, balance changes, duty reminders, and system events.

## Architecture Overview

```
Directus Flow (event/schedule trigger)
        │
        ├── Create notification in `notifications` collection
        │     └── useNotifications.ts polls every 20s → NotificationBell
        │
        └── Exec → Request → FastAPI POST /api/send-push
              └── pywebpush → Browser push → Service Worker
                    └── notificationclick → navigate to relevant page
```

## Collections Used

## Web Push

### Flow

```
1. User logs in → usePushNotifications.subscribe()
2. Browser prompt → user grants permission
3. POST /items/push_subscriptions (with dedup check by endpoint)
4. Directus Flow triggers → exec builds payload
5. Request operation → POST FastAPI /api/send-push
6. FastAPI fetches subscriptions by user_id
7. pywebpush sends to each endpoint
8. Service Worker shows system notification
9. User clicks → navigate to relevant page
```

### Infrastructure

| Component | Details |
|---|---|
| VAPID keys | Generated via web-push CLI, stored in .env |
| Push service | FastAPI with pywebpush library |
| Service Worker | generateSW strategy, imports push-handler.js |
| Subscription dedup | GET by endpoint before POST |
| Notification click | SW focuses existing tab or opens new one, navigates to URL |

### Key Fixes (Session 27.06.2026)

- `user` field must be passed explicitly in POST body for `push_subscriptions` — `$CURRENT_USER` preset only works for Directus UI, not API calls
- `navigateFallback: null` is required in workbox — without it the SW intercepts Directus admin navigation requests and breaks them
- Read permission required on `push_subscriptions` — `usePushNotifications` checks for duplicate endpoints before saving
- Subscription dedup: GET by endpoint first, skip if exists — prevents duplicate pushes on page reload

## State Machine Integration

Notifications are created by Directus Flows reacting to state changes:

```
cook_queue.status: scheduled → cooking → ready → completed
                              ↓           ↓
                    Cook Assigned     Lunch Ready
                    (on dish_name)    (on status=ready)

cook_queue.status: * → cancelled
                      ↓
                    Cook Cancelled
```

## Key Design Decisions

- **Polling over WebSocket** — `useNotifications` polls every 20s. WebSocket adds complexity not justified by current scale.
- **Flow-per-event pattern** — Each notification type has its own flow, not one monolithic flow. Easier to debug and maintain.
- **Utility flow for creation** — All flows delegate to `[Util] Create Notification` to avoid duplicate notification-creation logic.
- **Throw-based guard in schedule flows** — Schedule flows use `throw Error` instead of Condition operations to stop execution when no work is needed (e.g., no stale cooks).
- **Admin proxy for send-push** — FastAPI authenticates as Directus admin to read push_subscriptions for any user.

## Edge Cases & Limitations

- **Chrome push not working** — FCM (Firebase Cloud Messaging) requires a Firebase project. Low priority.
- **Morning Reminder uses throw Error** — If no cook is scheduled, the flow throws to stop. This logs as a flow error but is intentional behavior.
- **CRON changes require restart** — Directus node-cron caches schedules at process start. Changing a CRON expression requires `docker compose restart directus`.
- **Flows not version-controlled** — Flow definitions live in the database. Local changes must be manually recreated on production.
- **iPhone push requires PWA** — Safari browser tab does not support push; only works via "Add to Home Screen".
